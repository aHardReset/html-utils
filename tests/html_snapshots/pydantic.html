    Models

    \\n\\n

    The primary means of defining objects in pydantic is via models \\n(models are simply classes which inherit from BaseModel).

    \\n

    You can think of models as similar to types in strictly typed languages, or as the requirements of a single endpoint\\nin an API.

    \\n

    Untrusted data can be passed to a model, and after parsing and validation pydantic guarantees that the fields\\nof the resultant model instance will conform to the field types defined on the model.

    \\n
    \\n

    Note

    \\n

    pydantic is primarily a parsing library, not a validation library.\\nValidation is a means to an end: building a model which conforms to the types and constraints provided.

    \\n

    In other words, pydantic guarantees the types and constraints of the output model, not the input data.

    \\n

    This might sound like an esoteric distinction, but it is not. If you\\\'re unsure what this means or\\nhow it might affect your usage you should read the section about Data Conversion below.

    \\n

    Although validation is not the main purpose of pydantic, you can use this library for custom validation.

    \\n
    \\n

    Basic model usage¶

    \\n

    from pydantic import BaseModel\\n\\nclass User(BaseModel):\\n    id: int\\n    name = 'Jane Doe'\\n
    \\nUser here is a model with two fields id which is an integer and is required, \\nand name which is a string and is not required (it has a default value). The type of name is inferred from the\\ndefault value, and so a type annotation is not required (however note this warning about field \\norder when some fields do not have type annotations).\\n
    user = User(id='123')\\nuser_x = User(id='123.45')\\n
    \\nuser here is an instance of User. Initialisation of the object will perform all parsing and validation,\\nif no ValidationError is raised, you know the resulting model instance is valid.\\n
    assert user.id == 123\\nassert user_x.id == 123\\nassert isinstance(user_x.id, int)  # Note that 123.45 was casted to an int and its value is 123\\n
    \\nMore details on the casting in the case of user_x can be found in Data Conversion.\\nFields of a model can be accessed as normal attributes of the user object.\\nThe string \\\'123\\\' has been cast to an int as per the field type\\n
    assert user.name == 'Jane Doe'\\n
    \\nname wasn\\\'t set when user was initialised, so it has the default value\\n
    assert user.__fields_set__ == {'id'}\\n
    \\nThe fields which were supplied when user was initialised.\\n
    assert user.dict() == dict(user) == {'id': 123, 'name': 'Jane Doe'}\\n
    \\nEither .dict() or dict(user) will provide a dict of fields, but .dict() can take numerous other arguments.\\n
    user.id = 321\\nassert user.id == 321\\n
    \\nThis model is mutable so field values can be changed.

    \\n

    Model properties¶

    \\n

    The example above only shows the tip of the iceberg of what models can do. \\nModels possess the following methods and attributes:

    \\n
    \\n
    dict()
    \\n
    returns a dictionary of the model\\\'s fields and values; \\ncf. exporting models
    \\n
    json()
    \\n
    returns a JSON string representation dict(); \\ncf. exporting models
    \\n
    copy()
    \\n
    returns a copy (by default, shallow copy) of the model; cf. exporting models
    \\n
    parse_obj()
    \\n
    a utility for loading any object into a model with error handling if the object is not a dictionary;\\ncf. helper functions
    \\n
    parse_raw()
    \\n
    a utility for loading strings of numerous formats; cf. helper functions
    \\n
    parse_file()
    \\n
    like parse_raw() but for file paths; cf. helper functions
    \\n
    from_orm()
    \\n
    loads data into a model from an arbitrary class; cf. ORM mode
    \\n
    schema()
    \\n
    returns a dictionary representing the model as JSON Schema; cf. schema
    \\n
    schema_json()
    \\n
    returns a JSON string representation of schema(); cf. schema
    \\n
    construct()
    \\n
    a class method for creating models without running validation; \\ncf. Creating models without validation
    \\n
    __fields_set__
    \\n
    Set of names of fields which were set when the model instance was initialised
    \\n
    __fields__
    \\n
    a dictionary of the model\\\'s fields
    \\n
    __config__
    \\n
    the configuration class for the model, cf. model config
    \\n
    \\n

    Recursive Models¶

    \\n

    More complex hierarchical data structures can be defined using models themselves as types in annotations.

    \\n
    \\n
    \\n
    \\n
    from typing import List, Optional\\nfrom pydantic import BaseModel\\n\\n\\nclass Foo(BaseModel):\\n    count: int\\n    size: Optional[float] = None\\n\\n\\nclass Bar(BaseModel):\\n    apple = 'x'\\n    banana = 'y'\\n\\n\\nclass Spam(BaseModel):\\n    foo: Foo\\n    bars: List[Bar]\\n\\n\\nm = Spam(foo={'count': 4}, bars=[{'apple': 'x1'}, {'apple': 'x2'}])\\nprint(m)\\n#> foo=Foo(count=4, size=None) bars=[Bar(apple='x1', banana='y'),\\n#> Bar(apple='x2', banana='y')]\\nprint(m.dict())\\n"""\\n{\\n    'foo': {'count': 4, 'size': None},\\n    'bars': [\\n        {'apple': 'x1', 'banana': 'y'},\\n        {'apple': 'x2', 'banana': 'y'},\\n    ],\\n}\\n"""\\n
    \\n
    \\n
    \\n
    from typing import Optional\\nfrom pydantic import BaseModel\\n\\n\\nclass Foo(BaseModel):\\n    count: int\\n    size: Optional[float] = None\\n\\n\\nclass Bar(BaseModel):\\n    apple = 'x'\\n    banana = 'y'\\n\\n\\nclass Spam(BaseModel):\\n    foo: Foo\\n    bars: list[Bar]\\n\\n\\nm = Spam(foo={'count': 4}, bars=[{'apple': 'x1'}, {'apple': 'x2'}])\\nprint(m)\\n#> foo=Foo(count=4, size=None) bars=[Bar(apple='x...ss="n">_processed_at)\\n#> 2022-11-17 14:53:24.487087\\nprint(m._secret_value)\\n#> 4\\n
    \\n
    \\n
    \\n
    \\n

    \\n

    Private attribute names must start with underscore to prevent conflicts with model fields: both _attr and __attr__ \\nare supported.

    \\n

    If Config.underscore_attrs_are_private is True, any non-ClassVar underscore attribute will be treated as private:

    \\n
    \\n
    \\n
    \\n
    from typing import ClassVar\\n\\nfrom pydantic import BaseModel\\n\\n\\nclass Model(BaseModel):\\n    _class_var: ClassVar[str] = 'class var value'\\n    _private_attr: str = 'private attr value'\\n\\n    class Config:\\n        underscore_attrs_are_private = True\\n\\n\\nprint(Model._class_var)\\n#> class var value\\nprint(Model._private_attr)\\n#> <member '_private_attr' of 'Model' objects>\\nprint(Model()._private_attr)\\n#> private attr value\\n
    \\n
    \\n
    \\n
    \\n

    (This script is complete, it should run "as is")

    \\n

    Upon class creation pydantic constructs __slots__ filled with private attributes.

    \\n

    Parsing data into a specified type¶

    \\n

    Pydantic includes a standalone utility function parse_obj_as that can be used to apply the parsing\\nlogic used to populate pydantic models in a more ad-hoc way. This function behaves similarly to\\nBaseModel.parse_obj, but works with arbitrary pydantic-compatible types.

    \\n

    This is especially useful when you want to parse results into a type that is not a direct subclass of BaseModel.\\nFor example:

    \\n
    \\n
    \\n
    \\n
    from typing import List\\n\\nfrom pydantic import BaseModel, parse_obj_as\\n\\n\\nclass Item(BaseModel):\\n    id: int\\n    name: str\\n\\n\\n# `item_data` could come from an API call, eg., via something like:\\n# item_data = requests.get('https://my-api.com/items').json()\\nitem_data = [{'id': 1, 'name': 'My Item'}]\\n\\nitems = parse_obj_as(List[Item], item_data)\\nprint(items)\\n#> [Item(id=1, name='My Item')]\\n
    \\n
    \\n
    \\n
    from pydantic import BaseModel, parse_obj_as\\n\\n\\nclass Item(BaseModel):\\n    id: int\\n    name: str\\n\\n\\n# `item_data` could come from an API call, eg., via something like:\\n# item_data = requests.get('https://my-api.com/items').json()\\nitem_data = [{'id': 1, 'name': 'My Item'}]\\n\\nitems = parse_obj_as(list[Item], item_data)\\nprint(items)\\n#> [Item(id=1, name='My Item')]\\n
    \\n
    \\n
    \\n
    \\n

    (This script is complete, it should run "as is")

    \\n

    This function is capable of parsing data into any of the types pydantic can handle as fields of a BaseModel.

    \\n

    Pydantic also includes two similar standalone functions called parse_file_as and parse_raw_as,\\nwhich are analogous to BaseModel.parse_file and BaseModel.parse_raw.

    \\n

    Data Conversion¶

    \\n

    pydantic may cast input data to force it to conform to model field types,\\nand in some cases this may result in a loss of information.\\nFor example:

    \\n
    \\n
    \\n
    \\n
    from pydantic import BaseModel\\n\\n\\nclass Model(BaseModel):\\n    a: int\\n    b: float\\n    c: str\\n\\n\\nprint(Model(a=3.1415, b=' 2.72 ', c=123).dict())\\n#> {'a': 3, 'b': 2.72, 'c': '123'}\\n
    \\n
    \\n
    \\n
    \\n

    (This script is complete, it should run "as is")

    \\n

    This is a deliberate decision of pydantic, and in general it\\\'s the most useful approach. See \\nhere for a longer discussion on the subject.

    \\n

    Nevertheless, strict type checking is partially supported.

    \\n

    Model signature¶

    \\n

    All pydantic models will have their signature generated based on their fields:

    \\n
    \\n
    \\n
    \\n
    import inspect\\nfrom pydantic import BaseModel, Field\\n\\n\\nclass FooModel(BaseModel):\\n    id: int\\n    name: str = None\\n    description: str = 'Foo'\\n    apple: int = Field(..., alias='pear')\\n\\n\\nprint(inspect.signature(FooModel))\\n#> (*, id: int, name: str = None, description: str = 'Foo', pear: int) -> None\\n
    \\n
    \\n
    \\n
    \\n

    (This script is complete, it should run "as is")

    \\n

    An accurate signature is useful for introspection purposes and libraries like FastAPI or hypothesis.

    \\n

    The generated signature will also respect custom __init__ functions:

    \\n
    \\n
    \\n
    \\n
    import inspect\\n\\nfrom pydantic import BaseModel\\n\\n\\nclass MyModel(BaseModel):\\n    id: int\\n    info: str = 'Foo'\\n\\n    def __init__(self, id: int = 1, *, bar: str, **data) -> None:\\n        """My custom init!"""\\n        super().__init__(id=id, bar=bar, **data)\\n\\n\\nprint(inspect.signature(MyModel))\\n#> (id: int = 1, *, bar: str, info: str = 'Foo') -> None\\n
    \\n
    \\n
    \\n
    \\n

    (This script is complete, it should run "as is")

    \\n

    To be included in the signature, a field\\\'s alias or name must be a valid Python identifier. \\npydantic prefers aliases over names, but may use field names if the alias is not a valid Python identifier.

    \\n

    If a field\\\'s alias and name are both invalid identifiers, a **data argument will be added.\\nIn addition, the **data argument will always be present in the signature if Config.extra is Extra.allow.

    \\n
    \\n

    Note

    \\n

    Types in the model signature are the same as declared in model annotations, \\nnot necessarily all the types that can actually be provided to that field.\\nThis may be fixed one day once #1055 is solved.

    \\n
    \\n

    Structural pattern matching¶

    \\n

    pydantic supports structural pattern matching for models, as introduced by PEP 636 in Python 3.10.

    \\n
    \\n
    \\n
    \\n
    from pydantic import BaseModel\\n\\n\\nclass Pet(BaseModel):\\n    name: str\\n    species: str\\n\\n\\na = Pet(name='Bones', species='dog')\\n\\nmatch a:\\n    # match `species` to 'dog', declare and initialize `dog_name`\\n    case Pet(species='dog', name=dog_name):\\n        print(f'{dog_name} is a dog')\\n        #> Bones is a dog\\n    # default case\\n    case _:\\n        print('No dog matched')\\n
    \\n
    \\n
    \\n
    \\n

    (This script is complete, it should run "as is")

    \\n
    \\n

    Note

    \\n

    A match-case statement may seem as if it creates a new model, but don\\\'t be fooled;
    \\nit is just syntactic sugar for getting an attribute and either comparing it or declaring and initializing it.
